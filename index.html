Stadium Rental Suite

A dual-application system for running Pokémon Stadium tournaments

Client Repo Server Repo

Overview

This project consists of two coordinated applications that communicate through a shared MongoDB database. Players build and submit teams through the client app, while the server app processes inputs and drives tournament gameplay. As the tournament was played on unmodified Nintendo 64 hardware, player inputs are set in the form of strings representing game inputs that the host then enters on the console. Note that despite what the naming convention might imply, these apps do not use client/server architecture. It was just a useful way for me to conceptualize them.

Architecture

Player → Client App → MongoDB → Server App → Tournament Output

How It Works

  • Client app serves available party choices from the shared MongoDB cluster
  • Players create and save their party
  • Party choices are stored in the cluster
  • Players submit match inputs
  • Server reads and translates inputs into game actions

Tech Stack

  • C# / .NET
  • MongoDB
  • Desktop applications (client/server)

Why This Project

This system demonstrates multi-application architecture, shared-state coordination, and real-time-style input handling across separate programs.
